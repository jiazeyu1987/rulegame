📄 PRD（产品需求文档）模板
1. 项目背景

项目简介：一句话说明这个产品/功能要解决什么问题。

市场/用户痛点：为什么要做？用户当前遇到什么困难？

目标：通过这个产品/功能要达到什么效果？

2. 产品目标

核心目标（必须完成的核心功能）

次要目标（锦上添花，能增强体验但不是必需的）

不做的目标（明确范围边界，避免需求膨胀）

3. 用户画像与使用场景

目标用户：主要使用人群，典型角色。

使用场景：用户在什么场景下使用这个功能/产品。

用户价值：用户通过这个功能获得的收益或体验提升。

4. 用户流程 / 游戏流程

使用流程 / 游戏完整流程：

进入 →

关键步骤 →

系统响应 →

分支 →

结束 / 下一步

建议配 流程图 / 状态图，一目了然。

5. 功能需求
5.1 核心功能

列出必须实现的功能点（按优先级排序）。

5.2 交互与界面

界面布局要求

动画 / 过渡效果

按钮、输入框、弹窗等交互说明

5.3 数据存储与管理

存储位置（本地/服务端）

存储结构（字段、数据格式）

同步/恢复机制

5.4 错误与异常处理

可能出现的异常场景

系统如何处理（提示、回退、日志记录等）

6. 技术需求

技术栈（前端、后端、数据库等）

打包 / 部署方式

外部依赖（地图组件、脚本引擎等）

日志/监控方案

7. 非功能需求

性能（加载速度、延迟、并发数）

安全性（数据保护、存储安全）

兼容性（操作系统/浏览器/客户端）

稳定性（异常退出后的容错机制）

可扩展性（后续是否易于扩展）

8. 开发优先级 / 里程碑

P0（必须优先完成）：

功能 A

功能 B

P1（次要，重要但可以延后）：

功能 C

P2（可选功能）：

功能 D

9. 依赖与风险

依赖项（第三方库、素材、API、外部团队）

风险点（需求不确定、技术难点、上线风险）

风险应对方案

10. 验收标准

每个功能对应的 验收条件（Given-When-Then 形式最好）。

示例：

Given 用户已完成登录，When 输入正确的验证码，Then 进入主页。

11. 附录

相关文档链接（设计稿、流程图、接口文档）
